# Runtime model audit — v0.6.18

Дата: 2026-03-31

## Что проверено по фактическим файлам

Аудит выполнен сравнением `v0.6.16` и `v0.6.17`, а также просмотром runtime-файлов проекта.

### Что реально было до `v0.6.17`

- `Dockerfile` собирал frontend и отдавал **статический `dist` через Caddy**.
- `Caddyfile` работал как `file_server` для SPA и не проксировал собственный `/api/*` backend проекта.
- UI подключался к выбранному backend через `Setup`, а не к встроенному локальному Ubuntu service внутри этого репозитория.
- `src/api/ubuntuService.ts` уже существовал как **контракт и клиентская обвязка**, но не как доказанный встроенный server runtime этого проекта.

### Что было ошибочно сделано в `v0.6.17`

В `v0.6.17` в проект была навязана новая runtime-модель:
- добавлен `backend/`;
- добавлен `docker/start.sh`;
- добавлен `.env.example`;
- изменён `Dockerfile` под `frontend + python backend`;
- изменён `Caddyfile` под проксирование `/api/*` внутрь нового backend.

Эта ветка **не была заранее согласована как штатная архитектура проекта**, поэтому релиз нельзя считать правильным продолжением линии.

## Честный вывод

До появления отдельно подтверждённого Ubuntu server-side контура проект следует считать:

**frontend-first UI**, который:
- сам по себе остаётся статическим приложением;
- подключается к выбранному backend через `Setup`;
- не должен притворяться, что уже содержит встроенный scheduler, SQLite runtime, jobs engine и server-side provider SSL checks.

## Следствие для проверки Провайдеров

Проверка SSL сертификатов Провайдеров остаётся ожидаемой продуктовой функцией, но её нельзя считать закрытой только за счёт frontend-кода. Для честной реализации нужен отдельно подтверждённый server-side этап.

## Решение в `v0.6.18`

- вернуть проект к честной runtime-модели `v0.6.16`;
- зафиксировать ошибочность `v0.6.17`;
- продолжать cleanup интерфейса;
- проектировать provider SSL как отдельный Ubuntu server-side этап, а не как скрыто внедрённый runtime внутри релиза без согласования.
