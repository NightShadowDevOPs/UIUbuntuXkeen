# База запросов на изменения

Актуально для релиза: **v0.6.53**

## Запросы пользователя
- Продолжить проект здесь и начать делать backend.
- Объяснить, как backend будет запускаться.
- Делать релизы поэтапно и по ходу объяснять, что делать пользователю на его стороне.
- Постоянно актуализировать документацию.
- Фиксировать все шаги и все запросы на изменения.
- Не предлагать workflow с копированием архива на сервер для каждого обновления; перейти на стабильную папку `/opt/UIUbuntu` и `git pull` от клонированного репозитория.
- Перед выдачей релиза многократно прогонять install/update сценарии.
- После живого запуска backend исправить экранные баги интеграции, а не отправлять пользователя обратно в другой режим.
- Оставаться на backend `ubuntu-service` и развивать уже этот контур.
- Следующим шагом добавить реальные SSL-действия прямо из UI и убрать необходимость ручного `systemctl restart` после обновления backend.

## Реализация в v0.6.49
- backend стартует как отдельный Ubuntu service, а не как часть `mihomo` и не как `cgi-bin/api.sh`
- в репозиторий добавлен отдельный каталог `backend/` с готовым FastAPI + SQLite MVP
- добавлены install/dev-run сценарии и systemd unit template
- документация и transfer-файлы обновлены под новый backend-этап
- шаги работ и пользовательские запросы зафиксированы в `docs/worklog-db.md` и `docs/change-requests-db.md`

## Дополнение v0.6.49
- После первой реальной установки backend на сервере выявлен критический дефект install-flow.
- Требование пользователя: перед выдачей релиза многократно прогонять install/update сценарии, а не только собирать архив.
- В ответ на это `v0.6.49` исправил сценарий установки и подготовил стабильный server workflow через `/opt/UIUbuntu/app`.

## Дополнение v0.6.51
- После живого подключения backend пользователь показал, что экран `Хосты 3x-ui` всё ещё сваливается в `capabilities-http-404`, хотя backend уже подключён.
- Требование по сути: не ломать выбранный backend-режим и не заставлять пользователя возвращаться на fallback/direct, если проблема во фронтовой сборке URL.
- В ответ `v0.6.51` чинит именно frontend-маршрутизацию для standalone backend и сохраняет режим `ubuntu-service` как основной.

## Дополнение v0.6.52
- После подтверждения рабочего backend-контура пользователь прямо спросил, оставаться ли на `ubuntu-service`; решение зафиксировано: да, это основной режим.
- Следующий запрос: сделать реальные server-side действия на экране `Хосты 3x-ui`, а не просто держать backend подключённым в фоне.
- Отдельный operational-запрос: install/update backend не должен требовать ручного `systemctl restart` только для того, чтобы новая версия реально подхватилась.
- В ответ `v0.6.52` добавляет два явных SSL-действия в UI и исправляет install/update сценарий backend с обязательным restart уже внутри `install.sh`.

## Дополнение v0.6.53
- После живого перехода на `ubuntu-service` пользователь подтвердил новый дефект: в этом режиме не работает диаграмма трафика и связанные realtime-виджеты.
- Требование по сути: не возвращать пользователя на `direct`, а довести realtime-контур именно на standalone Ubuntu backend.
- В ответ `v0.6.53` добавляет WebSocket-маршруты и базовую host telemetry для `traffic`, `memory`, `connections`, `logs`, чтобы overview и traffic-экран перестали терять live-данные на `ubuntu-service`.
