# UIUbuntuXkeen v0.6.43 — backend contour audit

## Что подтверждено по факту
- UI обслуживается `mihomo.service`.
- Mihomo запускается как `/usr/local/bin/mihomo -d /etc/mihomo`.
- Папка UI на живом хосте: `/etc/mihomo/uiubuntu`.
- Подтверждённый recovery/update path: остановить `mihomo`, удалить папку UI, запустить `mihomo` снова, дождаться автоматической загрузки UI из репозитория.

## Что есть в проекте сейчас
- фронтенд UI на Vue/Vite;
- `api.sh` как существующий script-based contour/bridge в репозитории;
- клиентский контракт `src/api/ubuntuService.ts`, который умеет работать с `/api/*`, если такой contour реально доступен;
- разделы `Хосты 3x-ui` и `Пользователи` уже присутствуют в UI.

## Что НЕ подтверждено как существующий runtime
- наличие отдельного standalone Ubuntu service на сервере;
- готовое server-side хранение `3x-ui Hosts` и `Users` в живом backend-контуре;
- готовая серверная SSL-проверка 3x-ui панелей в текущем runtime без дополнительной реализации.

## Практический вывод
Следующий backend-шаг должен опираться не на новую сущность, а на фактически существующий server-side contour проекта. Сначала нужно отдельно разобрать:
- какие возможности уже есть в `api.sh`;
- какие capability-флаги реально возвращаются runtime;
- где безопасно хранить новые данные без появления нового фантомного service.
